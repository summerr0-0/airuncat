# 한 줄 정체성

> **airuncat = Harness Engineering을 위한 메뉴바 콕핏(cockpit).**
> 귀여운 세션 모니터가 아니라, **AI 코딩 하네스를 진단하고 키워주는 메뉴바 코치**.

# 왜 이 방향인가

airuncat은 세션 모니터(고양이 + 클릭 복귀)로 시작했지만, 탭이 늘며
(Skills / Prompts / MCP / Memory / Rules·Hooks·Permissions → Harness Score → 자동 세팅)
무게중심이 "**보는 도구**"에서 "**관리·개선하는 도구**"로 이동했다. 이건 표류가 아니라
더 강한 정체성을 찾은 것이다.

- `harness-checklist`의 5축 성숙도 방법론이 *이론*이라면, airuncat은 그걸
  **눈에 보이게(Score) · 실행 가능하게(자동 세팅) · 습관이 되게(메뉴바 상주)** 만드는 GUI다.
- 고양이는 "개발 인프라 도구"를 **귀엽고 늘 곁에 있게** 만드는 성격(soul).
- *메뉴바에서 Claude Code 하네스를 진단·정비* 하는 도구는 현재 없다 → **차별점(moat)**.

# 복리 루프 (airuncat이 소유할 것)

```
관찰(Stats·Sessions) → 진단(Harness Score) → 보완(자동 세팅) → 심화(Hook 레시피·rule 템플릿) → 다시 관찰
```

이것이 하네스 체크리스트 5축의 "개선(compounding)"을 도구로 구현한 것.
"51번째 작업이 50번째보다 낫다"를 airuncat이 대행한다.

# 로드맵

| Phase | 내용 | 상태 |
|-------|------|------|
| 13 | Harness Score — 5축 성숙도 진단(A~F) | 완료 |
| 14 | Harness 자동 세팅 — ✗ 항목 보완(CLAUDE.md/rule/deny권한 생성, hook 비활성 템플릿) | 완료 |
| **15** | **Hook 레시피 카탈로그** — 프로젝트 타입 인식, 큐레이팅된 안전한 훅을 골라 검토 후 활성화 | 기획 |
| 16 | 하네스 세팅 마법사 — 새 프로젝트를 F→B로 끌어올리는 한 흐름(CLAUDE.md→rules→permissions→hook) | 백로그 |
| 17+ | 내용 품질 진단(LLM, `/check-harness` 헤드리스), rule 템플릿 라이브러리 | 백로그 |

## Phase 15가 "훅 세팅"의 정답인 이유

Phase 14는 훅을 **빈 비활성 템플릿**으로만 만든다(임의 명령 자동 실행 위험). "훅도 세팅해줘"의
올바른 구현은 *AI가 명령을 추측해 주입*이 아니라 **레시피 카탈로그**다:
큐레이팅된 안전한 훅을 **고르고 → 실제 명령(프로젝트 타입에 맞춰 채워짐)을 보고 → 수정 → 켠다.**
검토 후 활성화 = 안전. Phase 14의 "빈 템플릿"이 "**검증된 원클릭 훅 모음**"으로 진화.

# 설계 원칙 (안 그러면 망가지는 지점)

1. **한 가지 일에 복무** — 모든 기능은 "네 하네스를 관찰→개선한다"에 기여해야 한다.
   아니면 쳐낸다. 탭마다 미니 앱이 되는 스코프 크리프를 경계.
2. **고양이를 묻지 마라** — 관리 UI가 늘어도 성격(ambient delight)이 중심에 남아야 차별점이 산다.
3. **신뢰가 생명** — "세팅해주는" 도구는 *진단 → 제안 → 사용자 확인 → 적용* 패턴을 법으로.
   파괴적·실행형(훅 명령 등)은 절대 자동으로 켜지 않는다. (Phase 13/14가 이미 지키는 원칙)
4. **세션은 읽기 전용** — `~/.claude/projects/**`는 관찰만. (`.claude/rules/read-only-sessions.md`)
